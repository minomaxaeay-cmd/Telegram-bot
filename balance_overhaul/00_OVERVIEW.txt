IRON DOMINION — COMBAT & BUILDINGS BALANCE OVERHAUL
MASTER OVERVIEW (00)
================================================================================

PURPOSE
-------
This folder defines the acceptance laws ("Mega-Goals", MGs) that the combat
system and the buildings system must jointly satisfy for the game to be
considered balanced, coherent, and fun. These are SPECIFICATION documents.
They describe WHAT must be true and HOW to verify it — they do not, by
themselves, change game code.

A Mega-Goal is considered COMPLETE only when every one of its binary criteria
(C1, C2, ...) passes its stated check. A criterion is BINARY: it either passes
or it fails — no partial credit.


THE 7 MEGA-GOALS
----------------
MG0  Prerequisites (FOUNDATION, not a balance goal)
       Establish a MAX_BUILDING_LEVEL and a complete gold-sink ledger so the
       horizon/economy goals become evaluable. See MG0_prerequisites.txt.

MG1  Parametric Backbone with Justified Exceptions
       Most unit/building stats come from one documented formula; every
       deliberate deviation is annotated with a design reason.

MG2  Honest Long Horizon (Respectful, not Deceptive)
       Progression is long and transparent: every play session yields visible
       progress, max is finite and disclosed, no hidden dead zones.

MG3  Fair Time-Gates (Fast Hook, Bounded Summit)
       Early power comes quickly; the top tier is gated by a sensible, tunable
       minimum time — never "impossible", never forced idle grind.

MG4  Closed Conservative Economy (No Inflation)
       Gold and troops are never created or destroyed except by declared
       sources (mines) and sinks (upkeep, casualties).

MG5  Sampled Non-Domination (Statistical, with Confidence)
       No army composition exceeds a win-rate ceiling in a large sampled
       tournament; every unit has a profitable counter at <= equal gold.

MG6  Buildings <-> Army Inseparable (Coherence)
       Neither system can be optimized alone; the joint optimum strictly
       dominates any economy-only or military-only build at equal spend.

MG7  Offline Self-Correcting Balance Harness (Design-Time, Never Live)
       A developer tool proposes constant adjustments from sim data and
       converges; live values are frozen within a season so players are never
       surprised mid-game.


REGRESSION RULE (READ THIS — IT IS THE CORE OF THE PROCESS)
-----------------------------------------------------------
The MGs are NOT independent. Fixing MG5 (combat balance) can easily break MG4
(economy conservation) or MG2 (progression curve). Therefore:

  1. Establish MG0 (prerequisites) FIRST, then work the Mega-Goals in order:
     MG0 -> MG1 -> MG2 -> ... -> MG7.
  2. A Mega-Goal may be ticked COMPLETE only when ALL of its own criteria pass.
  3. AFTER completing any Mega-Goal N, you MUST re-run the checks for ALL
     previously completed Mega-Goals (1 .. N-1).
       - If any earlier criterion now fails, Mega-Goal N is NOT complete.
       - Fix the regression, then re-verify 1 .. N again before moving on.
  4. Only when 1 .. N all pass simultaneously may you advance to N+1.

This guarantees that the final state satisfies all 7 Mega-Goals at the same
time, not one-at-a-time in isolation.


HOW CRITERIA ARE WRITTEN
------------------------
Each MG file contains:
  - LAW          : one-sentence statement of the principle.
  - TUNABLES     : named placeholder values (e.g. WINRATE_CEILING = 55%).
                   Set these to concrete numbers before verification.
  - CRITERIA     : C1, C2, ... Each has:
                     STATEMENT  - the binary condition.
                     CHECK      - exactly how to confirm pass/fail.
                     PASS WHEN  - the explicit pass condition.


GLOBAL TUNABLES (shared defaults; individual files may override)
----------------------------------------------------------------
  SIM_COUNT             = 1000     # Monte Carlo battles per matchup
                                   # (n=1000 -> 95% CI ~ +/-3.1% on a 50% rate;
                                   #  n=200 was +/-6.9%, too wide for a 5% band)
  WINRATE_CEILING       = 55%      # max allowed global win-rate for any strategy
  WINRATE_FLOOR_COUNTER = 60%      # min win-rate a valid counter must achieve
                                   # (50% is a coin-flip, not a real counter)
  CONFIDENCE            = 95%      # confidence interval for sampled claims
  STAT_FORMULA_TOL      = 10%      # +/- tolerance for "matches the backbone"
  FORMULA_COVERAGE      = 70%      # min % of stats that must match the backbone
                                   # (intentional RPS asymmetry => <100% by design)
  GOLD_RECONCILE_TOL    = 1 gold   # ledger reconciliation tolerance
                                   # (gold is a float; exact-0 fails on rounding)
  MAX_BUILDING_LEVEL    = 33       # LOCKED (MG0): caps the 1.6^lvl cost curve so
                                   # gold-mine max ~11.45 yr (> 10-yr gate). In
                                   # config.py. See MG0_prerequisites.txt.


VERIFICATION ARTIFACT (recommended)
-----------------------------------
Implement the checks as a runnable test/sim harness (e.g. a balance_tests
module) so that ticking a Mega-Goal is the result of an actual passing run,
not a judgement call. The MG7 harness should be reusable for MG5/MG6 sims.
DELIVERED: balance_sim.py (real-sim tournament/counter harness) and
balance_harness.py (MG7 self-correcting loop). Both need a Python environment.

CROSS-MG REGRESSION: later changes do not break earlier goals --
  - peasant hp stays 15 (MG5/real-sim): economy uses peasant COST (20, unchanged)
    so MG0/MG2/MG3/MG4 untouched; MG1 coverage 75% (exception justified by sim).
  - watchtower fort + blacksmith Forge (MG6): symmetric/bounded combat layers that
    add no gold flows (MG4 ok) and no unit-stat changes (MG1 ok); uniform so they
    preserve MG5 RPS; do not affect time-gates (MG3) or the cap (MG0).

REAL-SIM EXECUTION (resolves the earlier 'Python blocked' limitation):
The authoring web sandbox cannot run python directly, BUT the real game logic
was run SERVER-SIDE as Snowflake Python stored procedures
(IRON_DOMINION_SIM.PUBLIC.RUN_MG5 / PEASANT_SWEEP / RUN_MG6), driven via SQL.
This produced AUTHORITATIVE stochastic results and CAUGHT A REAL ERROR: a
deterministic SQL model had wrongly flagged a "peasant-screen deathball" at
hp 15 and prompted a nerf to hp 7; the real sim disproved it (hp 15 ceiling
~74%, hp 7 made peasant a dead unit), so the nerf was reverted. Lesson baked in:
tune combat only against the real sim. The same proc pattern can run the MG7
correction loop if desired. balance_sim.py / balance_harness.py remain as
local-Python equivalents.
